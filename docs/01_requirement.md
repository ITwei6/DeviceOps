## 1. 项目背景

### 1.1 为什么需要智能设备运维平台

随着机器人、工业设备、智能终端等设备在企业生产、仓储物流、园区管理和边缘计算场景中的大量部署，设备运维工作已经从单机维护逐步转向多设备、多地点、多系统协同的持续运营。设备运行过程中会持续产生状态数据、日志数据、异常事件和维修记录，这些数据如果不能被统一接入、分析和沉淀，运维团队很难及时判断设备健康状况，也难以形成可复用的故障处理经验。

DeviceOps 的目标是建设一个接近真实企业级场景的智能设备运维平台，通过设备接入、实时状态监控、异常告警、日志分析、知识库检索和 Agent 辅助诊断，帮助工程师更快发现问题、定位问题并形成诊断报告，从而降低设备停机时间和人工排查成本。

### 1.2 传统运维方式的问题

传统设备运维方式通常依赖人工巡检、分散日志、经验判断和临时沟通，主要存在以下问题：

- 数据分散：设备状态、运行日志、告警记录、维修记录分散在不同系统或文件中，缺少统一视图。
- 响应滞后：异常往往在业务受影响后才被发现，缺少实时监控和自动告警机制。
- 定位困难：故障原因可能涉及设备状态、通信链路、服务处理、历史缺陷和外部环境，人工排查链路长。
- 经验不可复用：故障处理依赖资深工程师经验，新人难以快速掌握，历史案例沉淀不足。
- 报告成本高：故障复盘和诊断报告需要人工整理日志、指标、处理过程和结论，效率较低。
- 规模扩展难：设备数量增加后，人工巡检和手工排查无法支撑企业级设备运维规模。

### 1.3 产品建设目标

DeviceOps 第一阶段的产品目标是定义智能设备运维平台的核心业务闭环：

```text
设备数据接入
↓
实时状态监控
↓
异常事件识别
↓
日志与历史故障查询
↓
AI 辅助故障定位
↓
诊断报告输出
↓
辅助工程师处置问题
```

## 2. 用户角色

### 2.1 运维工程师

#### 工作内容

- 监控设备在线状态、运行状态和异常告警。
- 处理设备故障、告警事件和现场反馈。
- 查询设备历史运行记录、日志和维修记录。
- 生成或提交故障处理记录。

#### 当前痛点

- 缺少统一设备状态看板，需要在多个系统之间切换。
- 告警信息不完整，无法快速判断告警严重程度和影响范围。
- 故障排查依赖人工查看日志，定位效率低。
- 历史故障案例不易检索，类似问题容易重复排查。

#### 使用本系统的需求

- 能够实时查看设备在线状态、关键指标和异常状态。
- 能够接收设备异常告警，并快速查看告警上下文。
- 能够按设备、时间、错误码、故障类型查询历史故障。
- 能够获得 AI 生成的故障分析建议和处理步骤。

### 2.2 开发工程师

#### 工作内容

- 开发和维护设备接入、数据处理、异常检测、日志分析等后台服务。
- 分析线上问题，定位服务异常、协议异常或数据异常。
- 根据运维反馈优化设备协议、告警规则和诊断逻辑。
- 支持故障复盘和系统优化。

#### 当前痛点

- 缺少统一的设备数据和服务日志关联能力。
- 运维反馈问题时，开发工程师需要手动追踪设备、服务和日志链路。
- 缺少结构化的故障上下文，问题复现和定位成本高。
- 告警规则和诊断逻辑缺少持续优化依据。

#### 使用本系统的需求

- 能够基于设备 ID、时间窗口和请求链路查询相关数据。
- 能够查看异常事件对应的原始数据、处理结果和日志信息。
- 能够分析告警规则命中情况和误报漏报情况。
- 能够将排查结论沉淀到知识库，供后续 Agent 诊断使用。

### 2.3 系统管理员

#### 工作内容

- 管理平台用户、角色和权限。
- 维护设备接入配置、系统参数和基础数据。
- 监控平台运行状态，保障平台可用性。
- 管理知识库、告警规则和数据留存策略。

#### 当前痛点

- 用户权限边界不清晰，容易出现越权访问或操作风险。
- 设备接入配置和系统参数分散，维护成本高。
- 缺少平台级运行状态和关键组件健康视图。
- 数据留存、日志检索和知识库内容缺少统一管理流程。

#### 使用本系统的需求

- 能够按角色管理用户权限。
- 能够配置设备基础信息、接入参数和告警规则。
- 能够查看平台服务健康状态和关键组件运行情况。
- 能够维护知识库内容，控制文档质量和可见范围。

## 3. 业务场景设计

### 3.1 场景一：设备异常报警

#### 场景描述

某台机器人在运行过程中温度持续升高，并上报异常错误码。运维工程师需要及时收到告警，并判断是否需要停机处理。

#### 输入

- 设备实时状态数据：设备 ID、电量、温度、速度、运行模式、在线状态。
- 设备异常数据：错误码、异常等级、异常发生时间。
- 告警规则：温度阈值、错误码等级、连续异常次数。

#### 系统处理

1. 系统接收设备上报的实时状态数据。
2. 状态监控模块更新设备最新状态。
3. 异常检测逻辑判断温度、错误码和运行状态是否满足告警条件。
4. 告警模块生成异常事件，记录设备上下文和触发规则。
5. 系统向运维工程师展示告警列表和设备详情。

#### 输出结果

- 新增设备异常告警。
- 告警详情包含设备 ID、异常类型、异常等级、触发时间、关键指标和建议动作。
- 设备状态看板中对应设备被标记为异常。

### 3.2 场景二：工程师查询历史故障

#### 场景描述

一台设备再次出现通信中断问题，工程师希望查询该设备过去是否发生过类似故障，并查看当时的处理记录。

#### 输入

- 查询条件：设备 ID、时间范围、错误码、故障类型、关键词。
- 历史数据：异常事件、设备状态快照、日志记录、维修记录、诊断报告。

#### 系统处理

1. 工程师输入查询条件。
2. 系统检索历史异常事件和日志记录。
3. 系统按时间、相关度、故障类型聚合展示历史故障。
4. 工程师打开某条历史故障，查看当时的状态、日志、处理动作和结论。
5. 系统支持将历史案例作为后续故障分析参考。

#### 输出结果

- 历史故障列表。
- 单个故障详情，包括发生时间、异常表现、关键日志、原因分析、处理措施和结果。
- 可复用的故障案例信息。

### 3.3 场景三：AI 辅助故障定位

#### 场景描述

设备出现综合异常，单凭告警信息难以判断根因。工程师希望系统结合实时状态、历史故障、日志和知识库内容给出初步定位建议。

#### 输入

- 当前异常事件。
- 设备近期状态数据。
- 相关服务日志。
- 历史故障案例。
- 知识库文档，如设备手册、错误码说明、维修 SOP。

#### 系统处理

1. 工程师在异常详情中发起 AI 诊断。
2. Agent 诊断模块收集设备状态、异常事件和日志上下文。
3. 系统从知识库中检索相关错误码、维修案例和操作建议。
4. Agent 根据上下文生成故障原因候选、置信度说明和排查步骤。
5. 工程师确认或补充诊断结论。

#### 输出结果

- AI 诊断结论草案。
- 可能原因列表。
- 推荐排查步骤。
- 关联日志、历史案例和知识库依据。
- 可保存的诊断报告。

## 4. 功能需求

### 4.1 设备管理

#### 功能目标

统一管理设备基础信息、接入状态和设备分组，为状态监控、告警、故障分析提供基础数据。

#### 输入

- 设备 ID、设备名称、设备类型、型号、所属组织、安装位置。
- 接入配置，如协议类型、认证信息、数据主题或通道标识。
- 设备状态，如在线、离线、维护中、停用。

#### 处理流程

1. 管理员新增或维护设备基础信息。
2. 系统校验设备 ID 唯一性和必要字段完整性。
3. 系统保存设备资料并建立设备状态档案。
4. 设备上报数据后，系统关联设备基础信息和实时状态。

#### 输出

- 设备列表。
- 设备详情。
- 设备状态档案。
- 设备接入配置记录。

### 4.2 状态监控

#### 功能目标

实时展示设备在线状态和关键运行指标，帮助运维工程师快速识别异常设备。

#### 输入

- 设备实时数据：电量、温度、速度、运行模式、在线状态、错误码。
- 设备心跳数据。
- 设备基础信息。

#### 处理流程

1. 系统接收设备实时数据和心跳。
2. 系统更新设备最新状态。
3. 系统按设备、分组和状态聚合展示。
4. 系统识别离线、超阈值和异常状态。

#### 输出

- 实时设备状态看板。
- 设备关键指标趋势。
- 异常设备列表。
- 单设备状态详情。

### 4.3 日志管理

#### 功能目标

集中管理设备日志和平台服务日志，支持故障排查、历史追踪和诊断报告生成。

#### 输入

- 设备运行日志。
- 设备错误日志。
- 平台服务日志。
- 查询条件，如时间范围、设备 ID、日志级别、关键词。

#### 处理流程

1. 系统采集并存储日志数据。
2. 系统按时间、来源、级别和设备标识建立检索条件。
3. 工程师输入查询条件。
4. 系统返回匹配日志并支持关联异常事件查看。

#### 输出

- 日志检索结果。
- 日志详情。
- 与异常事件关联的日志上下文。

### 4.4 异常告警

#### 功能目标

基于设备状态、错误码和规则配置自动识别异常，生成可跟踪的告警事件。

#### 输入

- 实时设备状态数据。
- 错误码和异常事件。
- 告警规则，如阈值、等级、持续时间、连续次数。

#### 处理流程

1. 系统读取实时状态和错误码。
2. 异常检测逻辑匹配告警规则。
3. 满足条件时生成告警事件。
4. 系统记录告警状态，如未处理、处理中、已恢复、已关闭。
5. 运维工程师查看并处理告警。

#### 输出

- 告警列表。
- 告警详情。
- 告警状态变更记录。
- 告警统计数据。

### 4.5 故障分析

#### 功能目标

围绕一次故障聚合状态、日志、历史案例和处理记录，帮助工程师完成原因分析和复盘。

#### 输入

- 异常告警。
- 设备状态快照。
- 相关日志。
- 历史故障记录。
- 工程师处理备注。

#### 处理流程

1. 工程师进入故障详情页。
2. 系统展示故障发生前后的状态变化和日志上下文。
3. 系统关联类似历史故障。
4. 工程师记录排查过程、原因和处理措施。
5. 系统生成结构化故障记录。

#### 输出

- 故障详情。
- 原因分析记录。
- 处理措施记录。
- 故障复盘资料。

### 4.6 知识库管理

#### 功能目标

管理设备手册、错误码说明、维修 SOP 和历史经验文档，为工程师查询和 Agent 诊断提供知识来源。

#### 输入

- 知识文档：Markdown、文本、手册内容、维修说明。
- 文档分类：设备类型、故障类型、错误码、适用范围。
- 文档状态：草稿、已发布、已废弃。

#### 处理流程

1. 管理员或工程师上传或维护知识文档。
2. 系统记录文档分类、版本和适用范围。
3. 管理员审核并发布知识内容。
4. 系统支持按关键词、错误码、设备类型检索知识。
5. Agent 诊断时可检索相关知识片段。

#### 输出

- 知识文档列表。
- 知识文档详情。
- 检索结果。
- 可用于诊断的知识片段。

### 4.7 Agent 诊断

#### 功能目标

基于异常上下文、日志、历史案例和知识库内容，辅助工程师生成故障定位建议和诊断报告。

#### 输入

- 当前告警事件。
- 设备实时状态和历史状态。
- 相关日志。
- 历史故障案例。
- 知识库检索结果。
- 工程师补充信息。

#### 处理流程

1. 工程师选择告警或故障记录并发起 Agent 诊断。
2. 系统收集设备、状态、日志和历史上下文。
3. 系统检索知识库中相关文档和案例。
4. Agent 生成可能原因、依据、排查建议和风险提示。
5. 工程师确认、修改或驳回诊断建议。
6. 系统保存诊断报告。

#### 输出

- AI 诊断建议。
- 可能根因列表。
- 推荐排查步骤。
- 关联证据。
- 诊断报告。

## 5. 非功能需求

### 5.1 性能

- 实时状态数据应支持秒级更新，满足运维人员查看最新设备状态的需求。
- 告警生成应尽量接近实时，异常发生后应在可接受时间内出现在告警列表。
- 日志检索应支持按设备 ID、时间范围、日志级别和关键词快速查询。
- 系统应支持后续按设备数量、数据频率和日志规模进行横向扩展。

### 5.2 可靠性

- 设备数据接入、状态处理和告警生成应具备错误处理能力，避免单条异常数据影响整体服务。
- 关键事件，如告警、故障记录、诊断报告，应保证可追踪、可查询。
- 平台应支持服务异常后的恢复和重新处理机制。
- 系统应保留必要日志，便于排查平台自身问题。

### 5.3 扩展性

- 设备类型、错误码、告警规则和知识文档分类应支持扩展。
- 后续应支持新增更多设备协议、更多诊断工具和更多数据源。
- 功能模块应保持清晰边界，便于按微服务方式逐步演进。
- Agent 诊断能力应支持后续接入更多工具调用和知识来源。

### 5.4 安全性

- 系统应支持用户身份认证和角色权限控制。
- 不同角色应具备不同的查看、配置、处理和管理权限。
- 设备接入配置、认证信息和系统参数应避免明文暴露。
- 操作记录、告警处理和诊断报告修改应具备审计能力。

### 5.5 可维护性

- 需求、架构、协议、数据库和任务计划应通过文档持续维护。
- 告警规则、知识库内容和错误码说明应可配置、可更新。
- 系统日志和诊断信息应便于开发工程师定位问题。
- 平台应遵循统一代码规范、模块边界和测试要求。

## 6. MVP 范围

### 6.1 第一版本必须实现

MVP 版本聚焦“设备状态可见、异常可发现、故障可追踪、AI 可辅助”的最小业务闭环。

必须实现的范围：

- 设备管理：支持设备基础信息维护、设备列表和设备详情。
- 状态监控：支持展示设备在线状态和关键指标，包括电量、温度、速度、错误码。
- 设备模拟数据：支持模拟机器人设备产生基础状态数据。
- 异常告警：支持基于温度、错误码、离线状态的基础告警规则。
- 日志管理：支持按设备 ID、时间范围和关键词查询基础日志。
- 历史故障：支持保存和查询故障记录、处理措施和结论。
- 知识库管理：支持维护基础错误码说明、维修 SOP 和历史案例。
- Agent 诊断：支持基于单个告警事件生成初步故障原因和排查建议。
- 诊断报告：支持保存 AI 诊断结果和工程师确认后的结论。

### 6.2 第一版本暂时不实现

暂不纳入 MVP 的范围：

- 多租户组织体系和复杂权限模型。
- 大规模设备批量接入和跨区域部署。
- 复杂工单流转、审批流和 SLA 管理。
- 高级可视化大屏和自定义仪表盘。
- 移动端 App 和短信、电话等外部通知渠道。
- 自动远程控制设备、自动修复或自动下发危险操作。
- 多模型调度、复杂 Agent 编排和自主闭环决策。
- 完整生产级灾备、多活部署和容量治理方案。

### 6.3 MVP 验收标准

- 运维工程师能够查看设备列表、设备详情和实时状态。
- 系统能够接收或模拟设备状态数据，并生成基础异常告警。
- 工程师能够查询历史故障和相关日志。
- 系统能够基于告警、日志和知识库内容生成 AI 诊断建议。
- 工程师能够保存诊断报告，并将处理结论沉淀为历史记录。
